## 2. 優先建議功能（P0 — 核心缺口）

### 2.1 財報數據整合

**功能說明**  
整合公開資訊觀測站的季度財報，提供 EPS、ROE、毛利率、營業利益率、負債比等核心財務指標，讓使用者在籌碼分析之外、同步掌握企業基本面。

**操盤手視角**  
主力進場通常有基本面支撐，若只看籌碼而不看財報，很容易追到業績空洞的股票。財報數據是過濾候選股的第一道關卡。

**資料來源**
- 公開資訊觀測站 MOPS：`https://mops.twse.com.tw`（HTML 爬取 or XBRL API）
- TWSE opendata：`https://openapi.twse.com.tw`（部分財務欄位）

**功能細項**

| 指標 | 說明 |
|------|------|
| EPS（每股盈餘） | 近 4 季 / 年化，YoY 漲幅 |
| ROE（股東權益報酬率） | 判斷資本配置效率 |
| 毛利率 / 營業利益率 | 判斷定價能力與成本控制 |
| 營收月增率 / 年增率 | 追蹤月營收趨勢（每月 10 日公告） |
| 負債比率 | 財務健全度評估 |
| 股利政策 | 現金股利 / 股票股利，近 5 年歷史 |

**前端呈現**  
個股頁新增「財報」分頁，以折線圖呈現 EPS 趨勢、長條圖呈現各季毛利率，並在股票列表加入財報篩選欄位。

---

### 2.2 除權息資訊與還原 K 線

**功能說明**  
記錄每支股票的除息日、除息金額、填息進度，並以還原價格（Adjusted Close）重繪 K 線圖，消除除息日因配息造成的跳空缺口。

**操盤手視角**  
未還原的 K 線會讓均線、支撐壓力位出現計算偏差，導致判斷錯誤。除息日附近也是法人操作的重要節點。

**資料來源**
- TWSE：`/exchangeReport/TWT48U`（上市除權息資訊）
- TPEx：TPEX openapi（上櫃）

**功能細項**

| 功能 | 說明 |
|------|------|
| 除息日行事曆 | 顯示未來 30 天內即將除息的股票清單 |
| 還原 K 線切換 | K 線圖右上角切換「原始 / 還原」模式 |
| 填息進度追蹤 | 除息後累積漲幅 vs 需填息金額，計算填息進度 % |
| 殖利率計算 | 以近 1 年配息 / 當前股價即時計算 |

---

### 2.3 多指標技術分析顯示

**功能說明**  
目前技術面篩選只有「突破高點」邏輯，需補充常用技術指標：MA、RSI、MACD、KD、布林通道等，直接疊加在 K 線圖上呈現。

**操盤手視角**  
MA 均線是進出場的核心依據。RSI 判斷超買超賣。MACD 黃金交叉是最普遍使用的趨勢確認信號。KD 是台股操盤手最習慣的短線工具。

**功能細項**

| 指標 | 計算邏輯 |
|------|----------|
| MA 5 / 10 / 20 / 60 / 120 / 240 | 簡單移動平均，可自訂顯示 |
| RSI（14 日） | Wilder 平滑法，超買 > 70、超賣 < 30 |
| MACD（12/26/9） | EMA 差離，黃金交叉 / 死亡交叉信號 |
| KD（9 日） | 隨機指標，K < 20 超賣、K > 80 超買 |
| 布林通道（20/2σ） | 判斷股價相對波動極端值 |
| 成交量 MA | 量能是否放大 or 萎縮 |

**後端實作**  
在 `/api/stocks/{symbol}/prices` 回傳資料時，後端直接帶 indicator 欄位（API 層即時計算），或新增 `/api/stocks/{symbol}/technicals?indicators=ma5,rsi,macd` 端點。

### 2.4 進階選股篩選器（Multi-Condition Screener）

**功能說明**  
讓使用者以拖拉介面自由組合多個條件（AND / OR）進行全市場選股，例如「外資連續買超 3 日 AND RSI < 40 AND 融券餘額下降」。

**操盤手視角**  
這是日常選股工作流最核心的工具。技術面 + 籌碼面 + 法人面的交叉驗證是找到強勢主流股的關鍵。

**可用條件類別**

| 類別 | 條件範例 |
|------|----------|
| 技術面 | RSI 範圍、均線多頭排列、量比（今量 / 均量）、創 N 日高 |
| 法人面 | 外資連續買超 N 日、投信買超張數 > N |
| 融資券面 | 融資餘額下降（籌碼鎖定）、融券大增（軋空機會） |
| 主力面 | 主力買超連續 N 日、特定券商在列 |
| 財報面 | EPS 年增率 > N%、ROE > N% |
| 籌碼面 | 大戶持股比率上升、散戶比率下降 |

**前端呈現**  
「選股工廠」獨立頁面，類似 TradingView Screener 的 UI，提供條件卡片組合、儲存策略命名、匯出 CSV。

## 3. 進階分析功能（P1 — 高價值）

### 3.1 籌碼評分系統（Chips Score）

**功能說明**  
將三大法人、融資券、主力進出、持股分佈等多維度籌碼資數，合成一個 0 ~ 100 的「籌碼評分」，快速判斷某檔股票的籌碼面是否健康。

**評分維度**

| 維度 | 權重 | 正向信號 |
|------|------|----------|
| 三大法人合計 | 30% | 外資 + 投信 + 自營合計買超 > 0 |
| 融資券比 | 20% | 融券增加（潛在軋空）或融資下降（籌碼乾淨）|
| 主力集中度 | 20% | 前 10 大買超券商合計佔交易量 > 閾值 |
| 持股大戶比率 | 20% | 1000 張以上大戶持股比 上升 |
| 券商勝率 | 10% | 歷史勝率高的主力券商出現在買超榜 |

**前端呈現**  
個股頁面「籌碼評分」圓形分數卡，股票列表加入評分排行欄位並可排序。

---

### 3.2 產業族群資金流向

**功能說明**  
以產業別（電子零組件、半導體、金融、傳產…）彙總每日三大法人買賣超，呈現「資金正在流入哪個產業」，輔助判斷市場主流輪動方向。

**操盤手視角**  
不看個股，先看產業輪動。法人資金集中流入某產業通常持續 3~5 個交易日，是搭轎的最佳時機。

**功能細項**

- 各產業當日 / 5 日 / 20 日法人買賣超金額長條圖（橫向排列，綠漲紅跌）
- 熱力圖（Heatmap）：產業 × 日期矩陣，顏色深淺代表資金強度
- 點擊產業：展開該產業底下的個股，並依法人買超排序

---

### 3.3 主力成本區估算

**功能說明**  
根據主力券商建倉日與交易量，估算主力的「均攤成本區間」，在 K 線圖上以色帶標示，判斷主力是否浮虧（可能護盤）或獲利（可能出貨）。

**計算方式**  
以主力事件（`broker_trade_events` 表）中尚未出場的建倉記錄，按交易量加權平均計算持有成本，並加 ±5% 形成「成本帶」。

**操盤手視角**  
主力不會輕易讓自己大幅虧損，當股價跌至成本區時是觀察護盤買盤的關鍵時機。

---

### 3.4 借券賣出深度分析

**功能說明**  
現有架構已有借券資料，但未完整呈現。需加強以下分析：
- 借券餘額趨勢圖（每日歷史）
- 借券賣出量 vs 融券放空量對比（兩種空頭力道比較）
- 借券大增的股票預警清單（偵測法人提前空頭布局）

**操盤手視角**  
借券是外資空頭最主要的工具，借券大增往往領先股價下跌 5~10 個交易日，是重要的預警信號。

---

### 3.5 週 K / 月 K 圖表

**功能說明**  
目前只有日 K，需新增週 K 與月 K 切換功能。可利用 PostgreSQL 的視窗函數在後端生成，或前端依日 K 資料聚合。

**實作建議**  
後端新增 `/api/stocks/{symbol}/prices?period=weekly|monthly` 端點，使用 SQL `DATE_TRUNC` 聚合：
```sql
SELECT
  DATE_TRUNC('week', date) AS time,
  FIRST(open, date)        AS open,
  MAX(high)                AS high,
  MIN(low)                 AS low,
  LAST(close, date)        AS close,
  SUM(volume)              AS volume
FROM daily_prices
WHERE symbol = $1
GROUP BY 1
ORDER BY 1;
```

## 4. 策略工具功能（P2 — 操盤輔助）

### 4.1 歷史回測引擎（Backtesting）

**功能說明**  
讓使用者將篩選策略應用在歷史資料上，評估該策略在過去的績效（年化報酬、最大回撤、勝率、平均持有天數）。

**操盤手視角**  
在投入實際資金前，先用歷史資料驗證策略的邏輯是否有效，避免在「感覺很對」但實際無效的策略上浪費時間。

**功能細項**

| 參數 | 說明 |
|------|------|
| 進場條件 | 從選股篩選器選擇（如：外資買超 + RSI < 40） |
| 出場條件 | 固定持有 N 日 / 停利 N% / 停損 N% |
| 回測期間 | 自訂起訖日期（建議最少 1 年） |
| 資金配置 | 等權 / 按訊號強度加權 |
| 統計結果 | 交易次數、勝率、平均獲利、最大回撤、夏普值 |

**後端實作**  
Go goroutine 並行跑每支股票的回測，結果快取至 Redis（因為計算量大），前端以非同步 + WebSocket 顯示進度。

---

### 4.2 支撐壓力位自動識別

**功能說明**  
在 K 線圖上自動標示關鍵支撐位與壓力位，來源包含：
- 近期波段高低點
- 成交量密集區（Volume Profile）
- 均線（MA20、MA60）當前價位

**操盤手視角**  
手動畫線花費大量時間，系統若能自動識別 3~5 個關鍵價位，每天盤前 10 分鐘掃一遍自選股就能完成布局評估。

**算法建議**  
使用 **ZigZag 波段識別算法**：找出局部極值（上下 N 個 bar 內最高 / 最低），聚類相近價位形成「支撐壓力區」，再加入成交量加權評分（交易越密集的價位，支撐壓力越強）。

---

### 4.3 相關性與強弱比較

**功能說明**  
輸入一支股票，自動找出同期走勢最相關的個股 / 類股，並提供「強弱對比圖」（同時段漲跌幅比較，以基準日為 0%，呈現相對表現）。

**操盤手視角**  
找到同族群中漲勢最強的「領頭羊」，觀察其同族群的「補漲股」，是產業輪動交易的核心操作手法。

**實作建議**  
後端預計算各股 60 日漲跌幅序列，使用 Pearson 相關係數，每日收盤後批次跑一次，結果存到 DB 的 `stock_correlations` 表。

---

### 4.4 交易日誌與績效追蹤

**功能說明**  
讓用戶手動記錄買賣交易（無需串接券商 API），系統自動回填歷史 K 線資料，計算每筆交易的損益，以及整體帳戶的績效報告。

**功能細項**

| 功能 | 說明 |
|------|------|
| 交易記錄輸入 | 日期、股票代碼、買賣方向、價格、張數 |
| 損益計算 | 含手續費（0.1425%）、交易稅（0.3%）的淨損益 |
| 持倉庫存 | 目前持有庫存與未實現損益（依即時報價更新）|
| 月績效報告 | 每月交易次數、勝率、損益，以圖表呈現 |
| 對比指標 | 與加權指數（大盤）的相對報酬比較 |

## 5. 市場感知功能（P3 — 情境判讀）

### 5.1 大盤與期貨市場指標

**功能說明**  
整合加權指數（TAIEX）、期貨未平倉量、大額交易人部位，提供市場整體溫度計。

**資料來源**

| 資料 | 來源 |
|------|------|
| 加權指數 OHLCV | TWSE openapi |
| 台指期未平倉量（OI） | TAIFEX（臺灣期貨交易所）opendata |
| 大額交易人淨部位 | TAIFEX `https://www.taifex.com.tw/cht/3/largeTraderFutDown` |
| PCR（Put/Call Ratio） | TAIFEX 選擇權日報表 |

**功能細項**

| 指標 | 操盤意涵 |
|------|----------|
| 台指期未平倉量 | OI 增加 + 期貨上漲 = 多方增倉，市場樂觀 |
| 大額交易人淨多 / 淨空 | 外資法人期貨方向，是最強的市場方向指標 |
| PCR（Put/Call Ratio）| PCR > 1.5 為極度悲觀（底部訊號）；< 0.5 為極度樂觀（頂部訊號）|
| 到期日效應追蹤 | 結算週前後市場異常波動提示 |

---

### 5.2 外資持股比率追蹤

**功能說明**  
TWSE 每月公告外資對各股的持股張數與持股比率（FINI 持股）。追蹤外資持股比率的長期趨勢，是判斷外資認同度的重要依據。

**資料來源**  
TWSE：`https://www.twse.com.tw/fund/MI_QIANW`（外資持股比率）

**功能細項**
- 個股頁面新增「外資持股比率」長期趨勢圖（月頻）
- 外資持股比率突破歷史新高的股票清單（視為外資持續認同）
- 接近持股上限（49% 或 100%）的股票警示（接近上限代表外資已無法繼續加碼）

---

### 5.3 法人自營避險比率分析

**功能說明**  
自營商的買賣超分為「自行買賣」與「避險」兩個子帳，合計才是對外公告數字。透過拆分，能更準確判斷自營商是真實看多還是只是 Delta 避險。

**資料來源**  
TWSE T86 API 欄位中已包含自行 / 避險分類。

**功能細項**
- 將自營商拆分為「自營自行」與「自營避險」分別呈現
- 當「自行」部分大幅超過「避險」時視為較強的看多信號

---

### 5.4 融資維持率警示

**功能說明**  
根據融資餘額與當前股價，計算個別股票的「融資維持率」。當維持率接近追繳警戒線（130%），系統發出預警，因為這代表後市可能出現強迫回補的賣壓。

**計算公式**

$$
維持率 = \frac{目前市值}{融資金額} \times 100\%
$$

**警示閾值**
- 維持率 < 166%：接近追繳（黃色警示）
- 維持率 < 130%：強制回補風險（紅色警示）

## 6. 通知與自動化功能（P4 — 效率提升）

### 6.1 條件警示系統（Alert System）

**功能說明**  
讓用戶設定條件（如：外資買超 > 1000 張、股價突破 MA20、籌碼評分 > 80），當條件觸發時主動推送通知。

**推送管道選項**

| 管道 | 難度 | 說明 |
|------|------|------|
| Line Notify | ⭐⭐ | 最簡單，用 Line Notify API |
| Telegram Bot | ⭐⭐ | 免費，設定 Bot Token + Chat ID |
| Email（SMTP）| ⭐ | 最通用，可用 Gmail SMTP |
| Web Push | ⭐⭐⭐ | 瀏覽器推播，需 Service Worker |

**後端實作**  
Go 排程每分鐘（盤中）或每日收盤後掃描所有警示條件，匹配則透過選定管道發送通知。

---

## 8. 技術實作建議

### 8.1 技術指標後端計算

建議新增 `/api/stocks/{symbol}/indicators` 端點，使用 Go 原生實作（無需第三方套件）：

```go
// MA 計算示例（後端 API 層即時運算，無需存 DB）
func calcMA(closes []float64, period int) []float64 {
    result := make([]float64, len(closes))
    for i := range closes {
        if i < period-1 {
            result[i] = 0
            continue
        }
        sum := 0.0
        for j := i - period + 1; j <= i; j++ {
            sum += closes[j]
        }
        result[i] = sum / float64(period)
    }
    return result
}
```

### 8.2 籌碼評分快取策略

每日收盤後 15:30 批次計算全市場籌碼評分，結果存入獨立欄位（或 Redis Hash），查詢時直接讀取，避免每次即時計算。

```sql
-- 新增欄位 or 獨立表
CREATE TABLE stock_chip_scores (
  symbol      VARCHAR(10) PRIMARY KEY,
  score       NUMERIC(5,2),
  breakdown   JSONB,    -- 各維度分數詳情
  calc_date   DATE,
  updated_at  TIMESTAMPTZ DEFAULT NOW()
);
```

### 8.3 選股篩選器設計原則

篩選器後端建議使用**動態 SQL 構建**（搭配白名單驗證防 SQL Injection），而非 ORM 條件鏈，以靈活支援使用者自訂組合：

```go
// 白名單驗證的安全欄位映射
var allowedColumns = map[string]string{
    "rsi":             "t.rsi_14",
    "foreign_net":     "i.foreign_net",
    "margin_balance":  "m.margin_balance",
}
```

### 8.4 通知架構

採用非同步事件驅動設計，避免通知發送阻塞主業務流程：

```
排程任務
  └─► 條件掃描（Go） ──→ Redis Pub/Sub（channel: alerts） ──→ Notification Worker（Go）
                                                               ├─► Line Notify API
                                                               ├─► Telegram Bot API
                                                               └─► SMTP Email
```
